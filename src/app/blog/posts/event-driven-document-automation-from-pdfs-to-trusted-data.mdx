Uploading a PDF is easy. Turning thousands of PDFs into trusted business records is a system design problem.

A real pipeline receives scans, rotated pages, duplicates, protected files, and documents in the wrong category. Fields may be handwritten, missing, or hidden in tables. Traffic can also arrive in bursts.

The safe goal is a controlled path from an untrusted document to a reviewed, traceable record, not a direct model-to-database call.

## The Production Problem

Consider a workflow that processes invoices, purchase orders, or insurance forms. Downstream teams may use the extracted data for payment, reporting, or compliance. A wrong supplier, amount, or policy number can cause more harm than a slow result.

The system therefore needs to answer several questions for every document:

- Was this exact file processed before?
- Which document type is it?
- What text, tables, and layout did the extractor find?
- Does the result match the expected schema?
- Which fields are uncertain or break business rules?
- Who approved a correction?
- Can the document be replayed after a code or model change?

These questions lead to an event-driven design with durable storage and explicit states.

## A Reference Architecture

```text
 [Upload API / Email / SFTP]
              |
              v
      [Private Object Storage]
              |
       object-created event
              v
        [Ingestion Queue] ---> [Dead Letter Queue]
              |
              v
       [Type Classifier]
              |
              v
      [OCR + Layout Service] <--- async completion event
              |
              v
      [Field Extraction Layer]
              |
              v
 [Schema + Business Rule Validator]
        |                 |
   high confidence    low confidence
        |                 |
        v                 v
 [Trusted Data Store] [Human Review Queue]
        ^                 |
        |------ approved--|

 Every stage writes status, version, evidence, and audit events.
```

Object storage keeps evidence. Queues absorb bursts. Extraction proposes a record. Validation and human review decide whether it is safe to publish.

## Start With Durable Ingestion

Put the original file in private object storage before AI work starts. Store it once, encrypt it, and assign a stable document identifier.

An ingestion record can include:

- `document_id`, generated by the platform
- `tenant_id`, taken from authenticated context
- object bucket, key, and immutable version
- SHA-256 content hash
- source channel and source reference
- upload time, media type, page count, and size
- current state and attempt count

A content hash detects duplicates but should not be the only key. The same file can belong to different tenants or cases. Combine tenant, workflow, source reference, and content hash.

Object events can arrive more than once. Claim the ingestion record with a unique constraint, a worker ID, and a lease expiry. The worker should renew the lease while it runs. A duplicate can return a safe no-op while the lease is active, but another worker must be able to take over an expired claim after a crash.

Do not acknowledge the queue message just because a claim exists. Mark the step complete and publish the next-stage event through a transactional outbox, then acknowledge. If the worker fails before that durable commit, the message becomes visible again and the expired lease allows recovery.

## Use the Queue as a Load Regulator

Extraction is slower than upload. A queue keeps ingestion responsive while workers run at a controlled rate.

Set worker concurrency below OCR service limits, retry throttling with backoff and jitter, and monitor queue age. When age rises, scale workers, slow non-urgent sources, or pause optional enrichment. Avoid retry storms.

Amazon Textract documents an asynchronous pattern for multipage files: start a job for a document in object storage, receive completion through a notification and queue, then fetch the paginated result. Its API also supports a client request token for idempotent starts. The same general pattern works with other OCR providers.

## Classify Before Extracting

A classifier should choose an extraction contract. Invoices, tax forms, and bank statements need different fields and rules.

Use approved classes plus `unknown`. Route uncertain files to review or a general path. A wrong class can produce convincing data under the wrong schema.

Preserve page numbers, tables, key-value links, bounding boxes, and confidence values. They support review and debugging.

Store raw OCR separately from normalized data so normalization can run again without repeating OCR.

## Make the Schema a Hard Boundary

The extraction layer converts OCR evidence into a document-specific object. A schema may require fields such as supplier name, invoice number, currency, line items, subtotal, tax, and total.

OpenAI Structured Outputs can constrain model output to a supplied JSON Schema. This is useful because valid structure removes a large class of parsing errors. It does not prove that a value is correct. A model can still place the wrong number in a valid numeric field.

Use several validation layers:

1. **Shape validation:** required fields, types, formats, and allowed values.
2. **Evidence validation:** every important field points to a page and source region.
3. **Business validation:** subtotal plus tax agrees with total, dates are plausible, and currency is supported.
4. **Reference validation:** supplier, account, or policy identifiers match trusted master data.
5. **Confidence policy:** critical fields meet field-specific thresholds.

Store `schema_version`, `extractor_version`, and `validator_version` with every result. Versions explain changed output during replay.

## Human Review Is a Designed State

Human review needs a queue with reasons, priority, ownership, and service targets.

Show the original page beside extracted fields, highlight the evidence, and explain the flag. Reasons can include low confidence, failed totals, or unknown suppliers.

Confidence policy should depend on risk. A low-confidence optional note may be accepted. A low-confidence bank account or payment amount should require review. Store both the machine proposal and the final human value. Never overwrite the proposal, because the difference is valuable for audits and future evaluation.

## Failure Modes and Decisions

| Failure mode | Unsafe response | Safer design |
| --- | --- | --- |
| Duplicate upload event | Run extraction twice | Claim a unique idempotency key and return the existing job |
| OCR provider throttles requests | Retry every message immediately | Limit concurrency and retry with backoff and jitter |
| Classifier chooses the wrong type | Force output into that schema | Support `unknown` and validate class-specific evidence |
| Valid JSON contains a wrong total | Publish because parsing passed | Apply arithmetic and master-data rules |
| Worker stops after writing data | Process again and create a duplicate | Use an atomic state transition or transactional outbox |
| Poisoned or unreadable PDF | Retry forever | Move it to quarantine with a reason and capped attempts |
| New extractor produces regressions | Replace trusted data in place | Write a new version and compare before promotion |

## Idempotency, Quarantine, and Replay

Give each stage a deterministic key such as `document_id + stage + version`, protected by a unique constraint. A worker claims the key, writes output, and records completion. Redelivery returns the stored result.

For a database publish, use an upsert keyed by tenant, business document ID, and record version. If publishing an event after the database write, use a transactional outbox. The record change and outbox event are committed together, then a separate relay sends the event. This closes the gap where a worker writes data but stops before notifying downstream systems.

Quarantine is for items that need a different action, not a place to forget failures. Store the original message, error class, safe error detail, attempt history, and next action. Separate temporary failures from permanent ones. A timeout may be retried. A corrupt PDF should wait for replacement or manual handling.

Replay should create a new processing run that references the same immutable source file. Operators select documents by time, tenant, failure reason, or version. The replay uses controlled batches and the same idempotency rules as live traffic. It should never require copying files into a special folder and hoping duplicates do not appear.

## Privacy, Security, and Audit

Documents often contain personal, financial, or legal data. Security belongs in the architecture:

- Encrypt object storage, queues, databases, and provider traffic.
- Use private network paths where supported.
- Give each worker access only to its required bucket prefix, queue, and table.
- Derive tenant identity from authentication, never from model output or document text.
- Use short retention for temporary OCR artifacts and logs.
- Redact document content from normal application logs.
- Scan uploads for malware and reject unsupported or oversized files.
- Record who viewed, changed, approved, replayed, or exported a document.

An audit event should include actor, action, document ID, run ID, old and new versions, timestamp, and reason. Sensitive values can remain in protected records while the audit stream stores references and change hashes.

## Implementation Checklist

- Define document classes, schemas, critical fields, and review rules.
- Store originals in private, versioned object storage.
- Create deterministic document and stage idempotency keys.
- Put a durable queue between ingestion and expensive processing.
- Set concurrency limits, retry caps, backoff, and a dead letter queue.
- Preserve OCR text, layout, confidence, and page evidence.
- Validate structure, evidence, business rules, and reference data.
- Build a reviewer screen with reason codes and side-by-side evidence.
- Use atomic state changes or a transactional outbox for publication.
- Design quarantine search and replay before production launch.
- Version schemas, prompts, models, validators, and normalized outputs.
- Test duplicate events, worker crashes, provider throttling, and deleted access.
- Monitor queue age, stage latency, review volume, retry rate, and validation failures.
- Apply retention, encryption, least privilege, tenant isolation, and audit controls.

## Conclusion

Reliable document automation is not a single OCR call or a clever prompt. It is a stateful data pipeline with durable evidence, controlled concurrency, strict validation, and clear recovery paths.

The most important design choice is to treat every extracted record as a proposal. Schema checks make the proposal usable. Business rules and source evidence make it believable. Human review makes uncertain cases safe. Idempotency, quarantine, and replay keep the system dependable when messages repeat or providers fail.

That architecture may look more involved than a direct model-to-database flow. It is also the difference between a demo that reads PDFs and a production system that creates trusted data.
